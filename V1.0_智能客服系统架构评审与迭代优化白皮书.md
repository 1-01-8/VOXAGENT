## 二、代码 Bug（需修复）

### Bug-1: CosyVoice 流式合成逻辑错误
- **文件**: `voice/cosyvoice.py` L140-148
- **问题**: `synthesize_stream()` 中 `_stream_gen()` 是同步生成器函数，但通过 `run_in_executor()` 调用只会返回生成器对象，不会迭代它。后续 `for audio_chunk in gen` 会在主事件循环中同步阻塞
- **影响**: 流式合成会阻塞事件循环，失去异步优势
- **建议修复**: 使用 `asyncio.Queue` 桥接：在 executor 中迭代生成器并 put 到 queue，async generator 从 queue 中 yield

### Bug-2: PermissionGuard 并发确认冲突
- **文件**: `agent/permission_guard.py` L46, L116
- **问题**: `_pending_confirm` 是单个 `asyncio.Future`。若两个 Level 2+ 工具同时请求确认，第二个会覆盖第一个 Future，导致第一个 await 永远不会 resolve
- **影响**: 并发确认场景下 Agent 挂起
- **建议修复**: 使用 `dict[str, asyncio.Future]` 按 tool_name 或 request_id 索引

### Bug-3: TransferPolicy 规则2 为死代码
- **文件**: `dialogue/transfer_policy.py` L97-98
- **问题**: `_check_rules()` 的规则2检查 `session.transfer_requested`，但调用方 `evaluate()` 在第69行已提前检查此条件并 return True。规则2永远不会被执行到
- **影响**: 无功能影响（代码冗余），但会误导维护者
- **建议修复**: 删除规则2的冗余检查

### Bug-4: asyncio.get_event_loop() 已弃用
- **文件**: `voice/sensevoice_stt.py` L109, `voice/cosyvoice.py` L93, `utils/session_logger.py` L70
- **问题**: Python 3.10+ 中 `asyncio.get_event_loop()` 已弃用，在非主线程中会报 DeprecationWarning 或抛异常
- **影响**: Python 3.12+ 环境下可能直接报错
- **建议修复**: 统一替换为 `asyncio.get_running_loop()`

### Bug-5: ReAct Action Input JSON 解析对嵌套 JSON 失效
- **文件**: `agent/react_agent.py` L247
- **问题**: 正则 `\{.+?\}` 使用非贪婪匹配，对嵌套 JSON（如 `{"data": {"key": "val"}}`）会在第一个 `}` 处截断，导致解析出不完整的 JSON
- **影响**: 当工具参数包含嵌套结构时 JSON 解析失败
- **建议修复**: 使用括号计数器或递归正则匹配完整的 JSON 块

## 三、逻辑漏洞（设计层面）

### Issue-1: 确认等待期间无法处理用户插话
- **文件**: `agent/permission_guard.py` L130-140
- **说明**: 等待 CONFIRM_RESPONSE 的 30 秒内，如果用户说了与确认无关的内容（如新问题），系统无法处理，只能等超时
- **建议**: 增加取消语义识别（"算了"/"不用了"→ confirmed=False）

### Issue-2: VIP 等级无填充机制
- **文件**: `dialogue/session.py` L60, `dialogue/transfer_policy.py` L105
- **说明**: `UserProfile.vip_level` 默认 0，V1 版本没有从 CRM 或工具调用中获取 VIP 等级的机制。TransferPolicy 的 VIP 规则永远不会触发
- **建议**: V1 可通过 GetCustomerInfoTool 的返回值填充 VIP 等级

### Issue-3: 文本模式下情绪检测未接入
- **文件**: `examples/customer_service_demo.py`
- **说明**: 纯文本模式下没有 STT 输出情绪标签，`EmotionDetector.update()` 从未被调用，情绪永远为 NEUTRAL。转人工的情绪规则在文本模式下失效
- **建议**: 文本模式可通过 LLM 分析用户文字情绪作为替代

### Issue-4: ReAct Scratchpad 无限增长
- **文件**: `agent/react_agent.py` L119-130
- **说明**: 每次迭代将完整 LLM 输出追加到 scratchpad。对于复杂任务（多次工具调用），scratchpad 可能超过 LLM 上下文窗口限制
- **建议**: 对早期 Observation 进行摘要，只保留最近 3 轮完整内容

### Issue-5: 长期记忆未实现持久化
- **文件**: `dialogue/memory_manager.py`
- **说明**: 中期摘要只存在内存中，会话结束后丢失。长期存储标注为 V2 对接 CRM，V1 版本用户历史信息无法跨会话保留
- **影响**: 回头客每次对话从零开始，无法提供个性化服务

### Issue-6: 质检敏感词过于宽泛
- **文件**: `utils/auto_qa.py` L26-29
- **说明**: "保证"/"一定"等词在客服场景中经常出现在正常回复中（如"保证金"/"一定会尽快处理"），当前粗暴匹配会产生大量误报
- **建议**: 改用短语级匹配或上下文感知的敏感词检测

### Issue-7: KBManager 直接访问 FAISSVectorStore 私有属性
- **文件**: `retrieval/kb_manager.py` L170, L202-216
- **说明**: `_rebuild_faiss()` 直接操作 `_metadata`, `_texts`, `_index` 等私有属性。如果 FAISSVectorStore 内部实现变更会直接破坏
- **建议**: 为 FAISSVectorStore 添加公共 API（如 `rebuild_without()`, `get_all_metadata()`）
