# VoxCareAgent 全链路低延迟 Debug 文档

> **版本**: v1.0.0 · 混合部署（本地 LLM + 云端语音）
> **日期**: 2026-04-20
> **作者**: VoxCare Infra
> **范围**: 将本地 ASR (SenseVoice/FunASR) + 本地 TTS (CosyVoice2) 迁移至 **SiliconFlow 云端 API**，
> LLM 与 Embedding 仍保持本地 Ollama，对全链路做端到端延迟审计与优化。

---

## 一、背景与动机（问题复盘）

### 1.1 旧架构（v0.x 纯本地）的瓶颈

| 组件 | 型号 | 实际表现 | 痛点 |
|-----|------|---------|------|
| LLM | Ollama + `qwen2.5:7b` | TTFT 200-400ms，40 tok/s | ✅ 正常 |
| Embedding | Ollama + `nomic-embed-text` | 单次 embed ~30ms | ✅ 正常 |
| STT | FunASR + `SenseVoice-Small` (cuda:0) | RTF ~0.05 | ✅ 正常但依赖 GPU 常驻 |
| **TTS** | **CosyVoice2-0.5B (cuda:0)** | **RTF 6-10，11 字要 14-20s** | ❌ **主瓶颈** |

旧架构下一次典型对话 (LLM 回 40 字 → 5 段 TTS) 的真实日志：
```
[PIPE-TIMER] LLM_FIRST_TOKEN  +4026ms
[PIPE-TIMER] seg#0 chars=11 tts_ms=20217 chunks=0   ← TRT 初始化 + 模型加载
[PIPE-TIMER] seg#1 chars=7  tts_ms=18860
[PIPE-TIMER] seg#2 chars=7  tts_ms=16244
[PIPE-TIMER] seg#3 chars=8  tts_ms=18667
[PIPE-TIMER] seg#4 chars=7  tts_ms=16753
[PIPE-TIMER] LLM_DONE wall=91012ms chars=40 segs=5
```

单轮对话耗时 **91s**，其中 TTS 占 90s。根因：
1. 每次 `synthesize()` 重新走 `_ensure_model()`（会话隔离的 consumer 实例）导致 JIT + wetext + TRT 反复初始化。
2. `libcudnn.so.8: cannot open shared object file` — `onnxruntime-gpu` 需要 cuDNN 8，当前 conda env 只有 cuDNN 9（PyTorch 内置），ORT 落回 CPU。
3. 启用 `load_trt=True` 触发 TRT 构建 `createInferBuilder Error Code 6: CUDA initialization failure with error: 35` — TensorRT 与 torch/onnxruntime 争抢 CUDA 上下文。
4. CosyVoice2 的原生 `stream=True` 实测 **RTF≈5.7**，比整段 `stream=False` (RTF≈0.97) 还慢 6×。

### 1.2 新架构（v1.0.0）的决策

- **LLM / Embedding 保持本地**：Ollama 已稳定，迁云无收益且会增加 API 成本。
- **ASR / TTS 迁 SiliconFlow 云端**：
  - 模型规格完全一致：`FunAudioLLM/SenseVoiceSmall` + `FunAudioLLM/CosyVoice2-0.5B`。
  - 云端已处理好 CUDA / cuDNN / TRT / vLLM 依赖，首字节 < 1s。
  - TTS 支持 HTTP chunked `stream=true`，可做真·流式推送。
- **保留降级路径**：provider 为可选；若云端不可达，仍可改回 `cosyvoice`/`sensevoice` 本地模式。

## 三、硅基流动 API 对接细节

### 3.1 ASR 端点

```
POST https://api.siliconflow.cn/v1/audio/transcriptions
Authorization: Bearer <VOR_SILICONFLOW_API_KEY>
Content-Type: multipart/form-data

file:  audio.wav (WAV 容器，16-bit mono PCM，任意采样率)
model: FunAudioLLM/SenseVoiceSmall
```
响应：`{"text": "..."}` — 仅文本，不含情绪标签（`emotion` 字段置为 `neutral`，兼容 `SenseVoiceResult` 结构）。

### 3.2 TTS 端点

```
POST https://api.siliconflow.cn/v1/audio/speech
Authorization: Bearer <VOR_SILICONFLOW_API_KEY>
Content-Type: application/json

{
  "model":           "FunAudioLLM/CosyVoice2-0.5B",
  "input":           "<待合成文本>",
  "voice":           "FunAudioLLM/CosyVoice2-0.5B:alex",
  "response_format": "pcm",
  "sample_rate":     24000,
  "stream":          true,
  "speed":           1.0
}
```
响应：`stream=true` 时为 **chunked transfer**，body 为原始 16-bit LE mono PCM。
我们在客户端按 4800 字节（≈100ms @ 24kHz/16bit）为一帧下推 WebSocket，前端 `AudioBufferSource` 按 `streamNextStart` 排队播放。

### 3.3 可选音色

预置 voice 后缀：`alex` / `anna` / `bella` / `benjamin` / `charles` / `claire` / `david` / `diana`。
如需自定义音色克隆，可用 SF 的 `/v1/uploads/audio/voice` 端点上传样本，但本期 v1.0.0 不引入。

## 四、全链路延迟审计

### 4.1 Pipeline 拓扑

```
[Browser Mic]
    │ (PCM 16kHz chunk)
    ▼
/ws/chat (WS)
    │  ①  WebSocket RTT   ~5ms (LAN)
    ▼
SiliconFlowSTT.transcribe_with_emotion
    │  ②  HTTP RTT + 推理 ~350-800ms (云端 SenseVoice)
    ▼
IntentRouter.classify            (本地 Ollama ~300ms)
    │  ③
    ▼
ReactAgent / LLM.stream          (本地 Ollama ~200ms TTFT, 40 tok/s)
    │  ④ token → 按标点切片
    ▼
tts_queue (asyncio.Queue)
    │
    ▼
SiliconFlowTTS.synthesize_http_stream(seg)
    │  ⑤  HTTP TTFB        ~400-800ms
    │      PCM chunk @100ms per frame
    ▼
WS audio_chunk → Browser AudioContext(24kHz) 排队播放
```

### 4.2 各段目标耗时

| 阶段 | v0.x 本地 | v1.0.0 混合 | 降幅 |
|-----|----------|------------|-----|
| STT（10s 音频） | ~500ms | ~600ms (含网络) | -20% (微劣) |
| LLM 首 Token | 400ms | 400ms | 持平 |
| TTS 首帧 (seg0) | **14-20s** | **~800ms** | **-95%** |
| 端到端 (用户说完话 → 听见第一个字) | **15-25s** | **~1.5-2.5s** | **-88%** |

### 4.3 关键优化点

#### ① 句级切分 + 流式 TTS
在 `web/server.py` 的 `_produce_from_stream` 里，LLM token 一遇到标点 (`，。！？；：,.!?;:\n`) 且缓冲 ≥ 6 字，立即丢入 `tts_queue`。这样：
- 首段合成在 LLM 还没写完第一句时就启动
- 多段 TTS 可以 **pipeline 并行**

#### ② HTTP Chunked 转 WebSocket Chunked
关键新增：`SiliconFlowTTS.synthesize_http_stream()` 直接 `async for` 云端响应字节流，每收到 4800 bytes 就 `ws.send_json({type: 'audio_chunk'})`。
前端 `enqueueAudioChunk` 用 `AudioContext.createBufferSource().start(startAt)` 精确排队，避免"一段一段等"的卡顿。

#### ③ httpx 连接池
`SiliconFlowSTT` / `SiliconFlowTTS` 均使用共享 `AsyncClient`，保持 TLS keep-alive。首次调用后，TLS 握手成本 (~100ms) 被摊平到整个会话周期之外。

#### ④ 预取窗口（兜底 synthesize_stream）
对于不支持 chunked 消费的调用方，`synthesize_stream` 以 `Semaphore(2)` 并发预取下一句，隐藏云端 RTT。

#### ⑤ 移除本地 GPU 初始化开销
v0.x 每个 WebSocket 会话内 `_tts_consumer` 会走 `_ensure_model()` 重新加载 CosyVoice2 + TRT + wetext → 每段多 3-5s 初始化。新架构无此开销。

## 五、全链路 Debug Checklist

### 5.1 启动前

- [ ] `VOR_SILICONFLOW_API_KEY` 已填入（登录 https://cloud.siliconflow.cn 生成 `sk-xxx`）。
- [ ] Ollama daemon 已运行: `ollama list | grep qwen2.5:7b`。
- [ ] FAISS 索引已导入: `ls data/faiss_index/` 不为空。

### 5.2 启动验证

```bash
uvicorn web.server:app --host 0.0.0.0 --port 8000
```
预期日志（前三行关键）：
```
INFO  SiliconFlow STT ready | model=FunAudioLLM/SenseVoiceSmall
INFO  SiliconFlow TTS ready | model=FunAudioLLM/CosyVoice2-0.5B voice=FunAudioLLM/CosyVoice2-0.5B:alex sr=24000 fmt=pcm
INFO  ✅ 服务已启动 | 语音模式 | LLM 已连接
```
注意：**不应再出现** `Converting onnx to trt...` / `libcudnn.so.8` / `CUDA initialization failure`。

### 5.3 运行时指标

浏览器 http://localhost:8000 发一条 "你好请介绍下你们的产品"，后端应看到：
```
[LLM-TIMER] wall=xxx ms prompt_eval=xxx gen=xxx tokens=xx tok/s=xx
[PIPE-TIMER] LLM_FIRST_TOKEN +300~500ms
[TTS-TIMER] sf_stream TTFB=400~900ms chars=10
[PIPE-TIMER] FIRST_AUDIO from_t0=1200~2500ms seg0_ttfb=400~900ms
[PIPE-TIMER] seg#0 chars=10 tts_ms=1200 audio_ms=1500 RTF=0.80 (stream)
...
[PIPE-TIMER] LLM_DONE wall=3500ms chars=40 segs=5
```

**验收阈值**：
| 指标 | 目标 | 故障判定 |
|-----|------|---------|
| LLM_FIRST_TOKEN | < 800ms | > 2s → 检查 Ollama GPU 利用率 |
| sf_stream TTFB  | < 1200ms | > 3s → 检查网络 / SF 配额 |
| FIRST_AUDIO from_t0 | < 2500ms | > 5s → 检查句切分最小阈值 `MIN_FLUSH_CHARS` |
| 每段 RTF | < 1.0 | > 1.5 → 云端拥塞，可切 `stream=false` 整段兜底 |

## 六、常见故障速查表

| 症状 | 原因 | 排查/修复 |
|-----|------|----------|
| `SiliconFlow TTS 调用失败: 401` | API Key 错/过期 | 重新在 SF 控制台生成 |
| `SiliconFlow TTS 调用失败: 429` | 并发或 RPM 超限 | 降 `prefetch=2→1`，或联系 SF 提升配额 |
| 前端音频 "咔咔" 断续 | `chunk_size` 过小或网络抖动 | 调大 `synthesize_http_stream(chunk_size=9600)` (=200ms) |
| 首帧仍然 > 3s | LLM 太慢把第一个标点吐出很晚 | 降 `MIN_FLUSH_CHARS=6→4`，更激进切片 |
| 整句读起来发音怪 | 过短片段被独立合成导致韵律断裂 | `MIN_FLUSH_CHARS` 调高到 10，或只在句号切（去掉逗号） |
| STT 空文本 | 上传 WAV header 错误 | 检查 `_pcm_to_wav` 的 `sample_rate`，必须等于前端 `sendAudio` 的采样率 |
| 云端 STT 返回乱码/英文 | `model=FunAudioLLM/SenseVoiceSmall` 默认自动语种，若需强中文可加 `language=zh` field | SF STT 当前 API 不开放 language，需上传足量中文内容 |
| WebSocket 收到 `sample_rate` 字段但前端仍 24k | 前端 hard-code 24kHz，强烈建议 SR 固定 24000 | 保持 `VOR_SILICONFLOW_TTS_SAMPLE_RATE=24000` |

## 七、回滚方案

单一环境变量即可切回本地：
```bash
# .env
VOR_STT_PROVIDER=sensevoice
VOR_TTS_PROVIDER=cosyvoice
```
所有本地代码（`cosyvoice.py` / `sensevoice_stt.py`）均未改动，向后兼容。
