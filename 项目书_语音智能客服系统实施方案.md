## 二、逻辑错误修正说明

以下是规划过程中发现的逻辑问题及修正：

### 错误 1：ASR 与情绪识别重复建模

**原计划：** Phase 1 用 FunASR Paraformer 做语音识别，Phase 2 再单独集成 SenseVoice-Small 做情绪识别。  
**问题：** 同时运行两个 ASR 级别的模型，推理资源翻倍，延迟叠加。  
**修正：** 直接用 **SenseVoice-Small** 替代 Paraformer 作为唯一 ASR 模型。SenseVoice-Small 单次推理同时输出：
- 转写文字（STT）
- 情绪标签（平静 / 疑惑 / 不满 / 愤怒）
- 事件标签（笑声 / 掌声等）

情绪检测模块无需独立模型，直接读取 STT 输出的 emotion 字段即可。

---

### 错误 2：TTS 与音色克隆文件冗余

**原计划：** 分别创建 `cosyvoice_tts.py` 和 `voice_cloner.py` 两个文件。  
**问题：** CosyVoice 2 是 TTS 和音色克隆的一体化模型，强行拆分造成接口割裂，参考音管理也需要两边同步。  
**修正：** 合并为单一文件 `voice/cosyvoice.py`，通过参数区分普通合成和克隆模式：
```python
# 普通合成
await cosyvoice.synthesize(text, mode="standard")
# 音色克隆
await cosyvoice.synthesize(text, mode="clone", reference_audio=ref_bytes)
```

---

### 错误 3：引用标注路径错误

**原计划：** 写 "扩展 `core/generator`"。  
**问题：** 框架中不存在 `core/generator` 模块，LLM 调用在 `llm/` 目录，生成回答在 `core/fast_talker.py` 的 `respond()` 方法内。  
**修正：** 引用标注逻辑在 `core/fast_talker.py` 的 `_format_context()` 方法中扩展，将来源 metadata 嵌入上下文格式，由 LLM Prompt 约束其在回答中标注引用编号。

---

### 错误 4：config.py 的 Literal 类型未同步

**原计划：** 直接新建 STT/TTS 文件，未提及配置层改动。  
**问题：** `config.py` 中 `stt_provider` 和 `tts_provider` 使用 Pydantic Literal 类型约束，若不更新约束，新提供商名称会在启动时触发 ValidationError。  
**修正：** Phase 1 必须同步修改 `config.py`：
```python
# 修改前
stt_provider: Literal["whisper", "deepgram", "openai"] = "whisper"
tts_provider: Literal["openai", "elevenlabs", "edge"] = "edge"

# 修改后
stt_provider: Literal["whisper", "deepgram", "openai", "sensevoice"] = "sensevoice"
tts_provider: Literal["openai", "elevenlabs", "edge", "cosyvoice"] = "cosyvoice"
```
同时在 `voice/stt.py` 和 `voice/tts.py` 的工厂函数 `create_stt()` / `create_tts()` 中注册新提供商。

---

### 错误 5：Agent 语音确认造成循环依赖

**原计划：** Level 2 工具"需用户语音确认"，未说明实现方式。  
**问题：** 如果 Agent 直接调用 TTS/STT 模块做确认，会形成 Agent → Voice → ConversationStream → Agent 的循环依赖，破坏架构分层。  
**修正：** 确认流程通过 **ConversationStream 事件** 实现：
1. Agent 执行 Level 2 操作前，发布 `CONFIRM_REQUIRED` 事件（携带操作描述）
2. `MemoryRouter` 监听该事件，触发 TTS 播报确认话术
3. 等待用户 STT 输入，发布 `CONFIRM_RESPONSE` 事件
4. Agent 订阅 `CONFIRM_RESPONSE` 事件，根据结果决定继续或取消

需在 `conversation_stream.py` 的 `EventType` 枚举中新增：
```python
CONFIRM_REQUIRED = "confirm_required"   # Agent 请求用户二次确认
CONFIRM_RESPONSE = "confirm_response"  # 用户确认结果（yes/no）
EMOTION_CHANGE = "emotion_change"       # 情绪状态变化
TRANSFER_REQUEST = "transfer_request"  # 触发转人工
```

---

### 错误 6：Phase 2 与 Phase 3 不必串行

**原计划：** Phase 2（对话管理）完成后才开始 Phase 3（RAG 强化）。  
**问题：** 两者无代码依赖关系，串行拉长了开发周期。  
**修正：** Phase 2 和 Phase 3 并行推进，分别由不同开发者负责，共用 5 周内完成。

## 三、修正后的实施计划

### Phase 1 · 语音底层升级（第 1 周）

**目标：** 替换 ASR / TTS 为中文优化模型，完成低延迟语音链路

| 编号 | 任务 | 改动位置 | 技术选型 | 延迟目标 |
|---|---|---|---|---|
| 1.1 | 新增 SenseVoice STT 实现 | 新建 `voice/sensevoice_stt.py` | SenseVoice-Small（STT + 情绪标签一体） | 首字 < 200ms |
| 1.2 | 注册到 STT 工厂函数 | `voice/stt.py` 的 `create_stt()` | 新增 "sensevoice" 分支 | — |
| 1.3 | 新增 CosyVoice TTS + 克隆实现 | 新建 `voice/cosyvoice.py` | CosyVoice 2（合成 + 零样本克隆一体） | 首字节 < 150ms |
| 1.4 | 注册到 TTS 工厂函数 | `voice/tts.py` 的 `create_tts()` | 新增 "cosyvoice" 分支 | — |
| 1.5 | 更新配置类型约束 | `voice_optimized_rag/config.py` | 扩展 Literal + 新增克隆配置项 | — |
| 1.6 | 升级 VAD 打断检测 | `voice/audio_stream.py` | 增加静音超时 + 用户打断立即停止 TTS | 实时 |

**新增文件：**
```
voice/
  sensevoice_stt.py   ← SenseVoice-Small 封装（STT + emotion tag）
  cosyvoice.py        ← CosyVoice 2 封装（合成 + 克隆，mode 参数切换）
```

**deliverable：** 可用中文语音输入、得到中文语音输出的完整回路，携带情绪标签。

### Phase 2 · 对话管理层（第 2～3 周，与 Phase 3 并行）

**目标：** 实现会话状态、意图路由、多轮记忆、转人工策略

| 编号 | 任务 | 改动位置 | 说明 |
|---|---|---|---|
| 2.1 | 新增事件类型 | `core/conversation_stream.py` | 增加 CONFIRM_REQUIRED / CONFIRM_RESPONSE / EMOTION_CHANGE / TRANSFER_REQUEST |
| 2.2 | 会话状态结构体 | 新建 `dialogue/session.py` | 包含 session_id、用户画像、槽位、情绪状态、任务状态 |
| 2.3 | 意图识别与路由 | 新建 `dialogue/intent_router.py` | 三路路由：知识问答型 → RAG / 任务执行型 → Agent / 闲聊型 → LLM 直答 |
| 2.4 | 情绪状态管理 | 新建 `dialogue/emotion_detector.py` | 读取 SenseVoice 输出的 emotion 字段，维护情绪状态机，无需独立模型 |
| 2.5 | 多轮记忆压缩 | 新建 `dialogue/memory_manager.py` | 短期 10 轮原文保留；超出后调用 LLM 压缩为摘要；会话结束写入长期存储 |
| 2.6 | 转人工策略 | 新建 `dialogue/transfer_policy.py` | 规则引擎：愤怒 2 轮 / 明确关键词 / Agent 连续 3 次失败 / VIP 请求 |

**新增目录：**
```
dialogue/
  __init__.py
  session.py            ← SessionContext 数据类
  intent_router.py      ← 意图分类 + 三路路由决策
  emotion_detector.py   ← 读取 SenseVoice 情绪标签，维护状态机
  memory_manager.py     ← 短期/中期/长期三级记忆管理
  transfer_policy.py    ← 转人工触发规则引擎
```

**依赖：** Phase 1 完成（需要 SenseVoice 的情绪标签输出）

### Phase 3 · RAG 知识层强化（第 2～3 周，与 Phase 2 并行）

**目标：** 在现有检索架构上增加销售场景专项能力

| 编号 | 任务 | 改动位置 | 说明 |
|---|---|---|---|
| 3.1 | 知识库分类与元数据标注 | `retrieval/document_loader.py` 扩展 | 按产品/政策/FAQ/竞品打分类标签存入 Qdrant payload，支持按类别过滤检索 |
| 3.2 | 口语化查询改写 | `core/slow_thinker.py` 扩展 `_predict_llm()` | 在 Prompt 中增加"口语 → 检索友好表达"改写指令 |

### Phase 4 · Agent 工具调用层（第 4 周）

**目标：** ReAct 模式自主决策 Agent，支持销售全业务工具调用

| 编号 | 任务 | 改动位置 | 说明 |
|---|---|---|---|
| 4.1 | ReAct Agent 主循环 | 新建 `agent/react_agent.py` | Thought → Action → Observation 闭环；复用 `llm/` 提供商，不另建 LLM 调用 |

| 4.3 | 查询类工具（Level 1） | 新建 `agent/tools/query_tools.py` | 查订单 / 查库存 / 查物流，只读无需确认 |
| 4.4 | 写操作工具（Level 2） | 新建 `agent/tools/write_tools.py` | 改地址 / 取消订单；执行前通过 ConversationStream 发布 CONFIRM_REQUIRED 事件等待用户语音确认 |

  base_tool.py          ← 工具抽象基类

  tools/
    __init__.py
    query_tools.py      ← 查询类（Level 1）
    write_tools.py      ← 写操作类（Level 2）
    finance_tools.py    ← 财务类（Level 3）

## 六、关键技术选型汇总

| 组件 | 选型 | 来源 | 说明 |
|---|---|---|---|
| ASR | SenseVoice-Small | 阿里达摩院 · FunASR | 中英日多语言，同时输出情绪标签，比 Whisper large 快 15x |
